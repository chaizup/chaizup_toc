When does each Formula work?

The system uses different formulas based on what the item is (Purchase vs Manufacture) and what day of the week it is.

F1
Always. It defines your "Safe Stock Level" (Target).
F2a/b
Always. It checks your "Current Reality" (Inventory Position).
F3
Always. It converts Reality vs Safety into a Red/Yellow/Green signal.
F7/F8
Sundays Only. It adjusts your Target if you were "Too Red" or "Too Green" all week.
🧭 Overview & Architecture 01

What it replaces: ERPNext ships with an "Auto Material Request" feature that fires when stock drops below a fixed reorder level. That model breaks for manufactured items, seasonal demand, and multi-warehouse operations. chaizup_toc replaces it entirely with Buffer Penetration % (BP%) — a dynamic signal that accounts for consumption rate, replenishment lead time, and demand variability.

Core idea: Every TOC-enabled item has a Target Buffer — the ideal stock level. The system continuously compares your Inventory Position against that target and colours the item Red / Yellow / Green. Red means order now; Green means you are well-covered.

💡
One number to watch: BP% (Buffer Penetration %). When BP% > 67% the item is Red — replenishment is urgent. When BP% < 33% it is Green — buffer is healthy. Everything in this system flows from that single percentage.

Data Flow

ERPNext data (Bin stock, Work Orders, Sales Orders, Delivery Notes) feeds into buffer_calculator.py every morning. The calculator produces BP% for every enabled item. mr_generator.py then creates replenishment documents for anything above Yellow. At the end of the week the DBM engine checks long-run patterns and grows or shrinks target buffers automatically.

🚀 Quick Start — First-Time Setup 02
⚠️
Follow this order exactly. Enabling items before warehouse rules are configured will produce inaccurate IP calculations — the system needs warehouse classification to know what counts as On-Hand vs WIP.
  • 1
    Classify your warehouses
    In TOC Settings → Warehouse Rules, tag each warehouse as Inventory (counts as on-hand stock), WIP (counts as work-in-progress for IP), or Excluded (ignored entirely). Without this classification every item falls back to summing all warehouses — usually wrong.
  • 2
    Set replenishment mode per item
    On each item in TOC Item Settings, set Auto Manufacture (creates Work Orders) or Auto Purchase (creates Purchase MRs), or leave both off for Monitor-only mode. You can bulk-apply mode using the multi-select Bulk Configure button.
  • 3
    Verify zone thresholds
    The defaults (Yellow start 33%, Red start 67%) are good for most companies. Adjust in TOC Settings → Zone Thresholds only if your lead times are highly predictable (tighten) or highly variable (loosen).
  • 4
    Configure MR generation
    In TOC Settings → MR Generation, set which zones trigger automatic Material Requests (default: Red + Yellow). Confirm the manufacture and purchase target warehouses. Enable auto_manufacture on items you want the system to automatically create Manufacture-type MRs for.
  • 5
    Enable items
    Open TOC Item Settings (bulk page) and filter TOC Not Active. Click each priority item, fill in RLT and VF per warehouse, save. The first ADU auto-calculation runs next morning; you can also enter a custom ADU immediately to get an instant first buffer.
  • 6
    Enable DBM after the first month
    Dynamic Buffer Management needs historical data to make sensible adjustments. Turn on TOC Settings → Enable DBM only after items have been active for at least one full Replenishment Lead Time cycle (usually 30 days).
📐 Formulas F1 – F8 03
F1 — Target Buffer
Target = ADU × RLT × VF
How much stock you need to hold to survive one full replenishment cycle under normal demand variability. This is your ideal stock level.
F2a — Inventory Position (Manufacture mode)
IP = On-Hand + WIP − Backorders
WIP = open Work Orders not yet received. Backorders = reserved_qty in Bin. For manufactured items we add pending production to available stock.
F2b — Inventory Position (Purchase mode)
IP = On-Hand + On-Order − Committed
On-Order = ordered_qty in Bin (open POs). Committed = WO Item unconsumed qty. For purchased items we add pending POs to available stock.
F3 — Buffer Penetration %
BP% = (Target − IP) / Target × 100
The main action signal. Negative BP% means you have more stock than target (very healthy). Above 100% means you are in stockout territory.
F4 — Order Quantity
Order Qty = Target − IP
How much to order or manufacture. Used as the qty on the Material Request. Zero or negative means no action needed.
F5 — Throughput / Constraint Unit (Manufacture mode)
T/CU = (Price − TVC) × Speed
Tie-breaker for the production priority board. Higher T/CU = more value per minute of constraint time. Automatically recalculated on Item save.
F6 — DAF Seasonal Adjustment
Adjusted = Target × DAF
Demand Adjustment Factor. 1.0 = no change. 1.3 = 30% extra buffer during peak season. Set per warehouse rule in the Item buffer rules table.
F7 / F8 — Dynamic Buffer Management
TMR: Target × (1 + adj%)
TMG: Target × (1 − adj%)
F7 grows the buffer when an item is Too Much Red (supply cannot keep up). F8 shrinks it when the item is Too Much Green (buffer is oversized). Runs every Sunday.

📦 Replenishment Modes — Manufacture / Purchase / Monitor 04

Each TOC-enabled item has exactly one replenishment mode, determined by two checkboxes on the item: custom_toc_auto_manufacture and custom_toc_auto_purchase. These are mutually exclusive — enabling both raises a validation error.

ModeHow setIP Formula (F2 — universal)ADU SourceAction at 07:00
Manufacture auto_manufacture = 1 OnHand + WIP + OnOrder − Backorders − Committed
All 5 components queried for every item regardless of mode. Unused sources return 0. 		All negative Stock Ledger Entries (outflows) Creates Manufacture MR → routes to Production Plan
Purchase auto_purchase = 1 All negative Stock Ledger Entries (outflows) Creates Purchase MR for the item (qty in purchase_uom)
Monitor Both = 0 All negative Stock Ledger Entries (outflows) No MR created — item tracked on dashboard only
✅
Industry-neutral: There is no FMCG-specific classification (FG/SFG/RM/PM). Any item can be Manufacture or Purchase mode regardless of item group or category. The IP formula is always universal — an item that is both occasionally purchased and manufactured will correctly reflect both WIP supply and OnOrder supply simultaneously. Use Bulk Configure in TOC Item Settings to apply mode to many items at once.
📐
UOM Standard: All IP components are in stock_uom (the item's canonical inventory unit from Item master). When creating a Purchase MR, the system automatically converts the shortage qty from stock_uom to purchase_uom using UOM Conversion Detail.conversion_factor. For example: stock_uom=Gram, purchase_uom=KG, cf=1000 → 5000 Gram order becomes 5 KG on the MR line.

ADU Calculation

ADU is computed universally from Stock Ledger Entries with actual_qty < 0 (any outflow from stock) over the configured period. This captures Delivery Notes, Stock Entries, Manufacturing consumption, and any other document that reduces stock — regardless of item category.

Warehouse-Aware Mode vs Fallback

If an item has buffer rules with explicit warehouse assignments, the calculator runs in warehouse-aware mode — it queries Bin data filtered to those warehouses, using your Inventory/WIP/Excluded classification. If no rules exist with warehouses, it falls back to summing all Bin rows for that item (useful during initial setup, not recommended long-term).

🚦 Zone System — Green / Yellow / Red / Black 05
ZoneBP% RangeMeaningAction
● Green Below 33% Buffer is healthy. Stock covers well beyond one RLT cycle. No action. Monitor for Too Much Green (DBM).
● Yellow 33% – 67% Buffer is being consumed. Plan replenishment now. MR generated automatically at 07:00. Confirm and action.
● Red 67% – 100% Buffer critically low. Stock may run out before next replenishment. Expedite. MR auto-generated. Use "Action Now" button on dashboard.
● Black Above 100% Stockout or near-stockout. IP is below zero buffer territory. Emergency. Stop production if no stock to consume. Escalate immediately.
🔴
Black zone means the buffer has fully penetrated. BP% > 100% means (Target − IP) > Target, which means IP is negative — you have promised more stock than you have or will have soon. Check the item immediately.

Configurable Thresholds

The 33% and 67% thresholds are defaults in TOC Settings. For items with very short, predictable lead times (e.g., a 1-day local supplier) you might tighten them to 20% / 50%. For highly variable imported materials you might loosen them to 40% / 75%.

⏰ Daily Operating Rhythm 06

All jobs run automatically. The schedule below assumes Asia/Kolkata timezone (UTC+5:30) — adjust server cron offsets if your ERPNext timezone differs.

06:00 daily
Sales Projection Automation
Finds the current calendar month's submitted Sales Projection. For each item line: computes net demand using Calc 1 or Calc 2, creates a Draft Work Order if production_capacity > 0 and an active BOM exists. Emails notifications to configured users.
06:30 daily
ADU Auto-Calculation
Recalculates Average Daily Usage for every enabled item where custom_adu = 0 (auto mode). Reads all Stock Ledger Entries with negative qty (outflows) over the configured period (30/90/180/365 days). Updates adu_value on each buffer rule row.
07:00 daily
MR Generation
Runs the full buffer calculator on all enabled items. Creates Material Requests for every item in Red or Yellow zone. Manufacture-mode items get Manufacture-type MRs routed to Production Plan. Purchase-mode items get Purchase MRs. Deduplication skips items that already have an open MR for the same warehouse.
07:30 daily
Procurement Monitor
Scans all Purchase-mode items in Red or Black zone and logs a warning to ERPNext Error Log. This is a monitor only — it does NOT create additional MRs. Use it to identify items where the 07:00 MR was created but supply has not responded.
08:00 daily
Buffer Snapshot (TOC Buffer Log)
Archives the current zone, BP%, IP, and target_buffer for every enabled item into TOC Buffer Log DocType. This historical data is used by the DBM engine to detect TMR and TMG patterns over time.
08:00 every Sunday
DBM Auto-Adjust (F7 / F8)
Reads the TOC Buffer Log archive. Applies TMR (F7) if an item has spent too many days in Red/Black zone over the past RLT period. Applies TMG (F8) if an item has been Green for multiple full RLT cycles. Updates target_buffer and logs the change with reason.
💡
Manual trigger: On the TOC Dashboard, click Action Now or Plan buttons to trigger MR generation for a specific replenishment mode without waiting for the 07:00 schedule. The button runs trigger_manual_run for all Red/Black/Yellow items of that type — not just the item you clicked.
📊 Sales Projection Automation 07

When a Sales Projection is submitted for the current calendar month, the engine automatically translates forecast lines into Draft Work Orders — bridging forecast → production planning without manual Production Plan creation.

Net Demand Formula

Net Required
Net = (projected_qty + prev_month_pending_SO)
    − curr_month_pending_SO
    − (current_stock + open_WO_remaining)
All SO filtering uses delivery_date (not transaction_date). WO qty = min(net_required, production_capacity). If net ≤ 0 or production_capacity = 0, skip.

SO Eligibility — Two Paths

PathConditionUse case
PATH A docstatus = 0 AND status IN configured workflow states Draft SOs that are approved but not yet submitted. Useful for companies that keep SOs in draft pending credit checks.
PATH B docstatus = 1 AND per_delivered_qty status IN configured pending statuses Submitted SOs that are not yet fully delivered. Default: "To Deliver and Bill", "To Deliver", "On Hold".
⚠️
Configure projection_pending_so_statuses in TOC Settings → Sales Projection Automation. If this field is empty the SQL IN () clause crashes MariaDB. The engine has a guard for this but always keep at least one status configured.

Post-PP 5-Step Flow (after WO creation)

  • 1
    Recursive BOM walk to find all sub-assemblies (max depth 10). Creates child WOs with custom_projection_parent_wo link.
  • 2
    Identifies purchased components from BOM. Checks if those components have existing open MRs (dedup).
  • 3
    Saves all Work Orders (docstatus=0, Draft). Stamping custom_toc_creation_reason with the formula breakdown.
  • 4
    Updates wo_status and wo_name on the Sales Projected Items child table row.
  • 5
    Stamps last_auto_run on the Sales Projection parent document. Sends notifications to configured users.
💡
Manual run: On a submitted Sales Projection for the current month, a Run Projection Automation button appears in the document toolbar. This calls run_projection_automation with your user as triggered_by. Only TOC Manager and System Manager roles can use it.
🔄 Dynamic Buffer Management (DBM) 08

Demand and lead times change over time. A buffer set in January may be too small in Diwali season and too large in February. DBM automatically grows and shrinks target buffers based on observed replenishment signals, so you do not need to manually retune every item.

F7 — TMR "Too Much Red"
Target × (1 + adj%)
Trigger: The item spent more than 20% of its last RLT window in Red or Black zone. Buffer is too small — demand or lead time increased.
Effect: Grows target buffer by adj% (default 33%). Recorded in tmr_count.
F8 — TMG "Too Much Green"
Target × (1 − adj%)
Trigger: The item stayed Green for 3 consecutive full RLT cycles. Buffer is oversized — tying up capital. Tracked via tmg_green_days.
Effect: Shrinks target buffer by adj% (default 33%). Never below 1-day ADU.

DBM Safeguards

SafeguardWhy it exists
Minimum buffer = 1 × ADUPrevents shrinking to zero during a slow sales month
Adjustment capped at 33% per runPrevents violent oscillation from a single outlier week
DBM disabled by defaultNew items need at least one RLT cycle of history before auto-adjust is meaningful
last_dbm_date tracked per itemPrevents double-adjustment if Sunday job runs more than once due to server issues
⚠️
Enable DBM only after 30+ days of data. If you turn it on day one, the engine will see that everything is Green (no demand history yet) and shrink all your buffers down to minimum — the exact opposite of what you want for a new item launch.
🔔 Real-Time Buffer Alerts 09

In addition to the morning scheduled job, the system sends instant alerts when a stock movement pushes a TOC-managed item into Red or Black zone.

Trigger Chain

Stock Ledger Entry — after_insert
Any stock movement (Delivery Note, Stock Entry, Purchase Receipt) creates a Stock Ledger Entry row. The after_insert hook checks if the affected item is TOC-enabled.
frappe.enqueue (enqueue_after_commit=True)
The hook enqueues check_realtime_alert with enqueue_after_commit=True — this ensures the alert runs only after the database transaction commits, so Bin.actual_qty is already updated when the alert fires.
check_realtime_alert (background job)
Re-calculates BP% for the item using the latest Bin data. If BP% >= the Red threshold, calls frappe.publish_realtime with event name toc_buffer_alert and item details.
Browser — frappe.realtime (socket.io)
Any open Frappe session subscribed to the event receives a toast notification. The TOC Dashboard auto-refreshes its data when a toc_buffer_alert event arrives.
💡
Why enqueue_after_commit? Without it, the background job might read Bin.actual_qty before the Delivery Note transaction commits — it would see stale stock data and miss the alert entirely.

🔧 Use Cases & Troubleshooting 13
Item not showing on dashboard

Check 1: Is custom_toc_enabled = 1 on the item?

Check 2: Does the item have at least one buffer rule row with a warehouse? Without a rule, it falls back to summing all warehouses — and may still show if ADU > 0.

Check 4: Is ADU zero? If custom_adu is off, wait for the 06:30 job or set a manual ADU value to get an immediate first buffer.

BP% is wrong / unexpected zone

Manufacture mode: Check that the WIP warehouse is classified correctly in Warehouse Rules. If a large Work Order sits in an Excluded warehouse, it will not count toward IP, making BP% artificially high.

Purchase mode: Check On-Order (Bin.ordered_qty). If a large PO was closed or cancelled without updating the Bin, ordered_qty may be stale. Run bench --site [site] execute erpnext.stock.utils.update_bin_qty to reconcile Bin data.

All modes: If ADU was last calculated a month ago and demand has changed, BP% will be based on stale ADU. Force recalculation via TOC Settings → Recalculate All Buffers.

Duplicate Material Requests being created

The MR generator has a dedup check: it calls _has_open_mr(item_code, warehouse) before creating. If you see duplicates:

  • Check if the existing MR is in Draft state (not submitted) — only open/draft MRs are checked.
  • Check if the warehouse on the existing MR matches the item's target warehouse in TOC Settings.
  • The ERPNext reorder override (reorder_override.py) should cancel ERPNext auto-MRs. Verify the override is active in hooks.py doc_events.
Sales Projection automation not creating Work Orders
  • Is enable_projection_automation = 1 in TOC Settings?
  • Is the Sales Projection submitted (docstatus = 1)?
  • Is the projection month = current calendar month? The engine only processes the current month.
  • Does each item have a Default BOM that is active and not expired?
  • Is production_capacity > 0 on the Sales Projected Items row? Zero capacity silently skips the item and sends a notification.
  • Check ERPNext Error Log for exceptions from projection_engine.py.
DBM is shrinking buffers unexpectedly

TMG fires when an item has been Green for 3 consecutive full-RLT cycles. This usually means the buffer was set too large at initial configuration.

Immediate fix: Disable DBM in TOC Settings, manually increase RLT or VF on the affected items, then re-enable DBM after the buffer normalises.

Root cause: Check if ADU dropped significantly (e.g., seasonal slowdown). If real demand is low, DBM shrinking is correct behaviour. Consider increasing VF instead of fighting DBM.

Dashboard shows Black zone but stock looks OK

Black = BP% > 100%, meaning IP < 0. This happens when:

  • Backorders (reserved_qty in Bin) exceed On-Hand + WIP. Someone reserved more stock than exists.
  • For Purchase-mode items: Committed qty (WO Item unconsumed) is very high relative to on-hand + on-order.
  • A warehouse classification error is excluding a large Inventory warehouse.

Always verify Bin data directly: frappe.get_all("Bin", filters={"item_code": "[item]"}, fields=["*"]) in the console to see raw actual_qty, ordered_qty, reserved_qty.

Page is blank after deploying changes to HTML template

Frappe caches Jinja templates in Redis. After any change to an .html template file you MUST flush the cache:

redis-cli -h redis-cache -p 6379 FLUSHALL

Also check browser console for SyntaxError — a raw single quote inside the template will cause a JS parse error because Frappe wraps templates in a JS single-quoted string. Replace all ' possessives with &apos;.
