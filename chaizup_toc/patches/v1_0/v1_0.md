All patches fix bugs from the initial v1.0 release. All are idempotent (safe to re-run).

## fix_date_filters.py

### Problem
Number Cards and Dashboard Charts were initially created by `setup/install.py` with date filters using `"Today"` as a raw string:

```json
{"filters_json": "[\"log_date\",\"=\",\"Today\"]"}
```

Frappe v14+ changed the filter format for temporal values. `"Today"` stopped working — the expected format is a tuple `["Timespan","today"]`. Affected cards showed `0` or a silent error.

### Fix
```python
for nc_name in ["TOC - Items in Red Zone", "TOC - Items in Yellow Zone", "TOC - Items in Green Zone"]:
    if frappe.db.exists("Number Card", nc_name):
        filters = frappe.db.get_value("Number Card", nc_name, "filters_json") or ""
        if '"Today"' in filters:
            new_filters = filters.replace('"log_date","=","Today"', '"log_date","Timespan","today"')
            frappe.db.set_value("Number Card", nc_name, "filters_json", new_filters)
```

Similar fix applied to `"TOC - Zone Distribution"` Dashboard Chart.

**Note**: This patch is superseded by the current `setup/install.py` which now uses `type="Custom"` + `method=` instead of date filters entirely — avoiding the Frappe version dependency. The patch still handles existing installations with the old filter-based cards.

## fix_old_field_refs.py

### Problem
Version 1.0 used a field called `custom_toc_generated` on Material Request. This was renamed to `custom_toc_recorded_by` with values "By System"/"By User". After the rename:
- Workspace shortcuts still filtered on `custom_toc_generated=1`
- Number Cards still filtered on `custom_toc_generated=1`

Both showed `0` (no records match the old field name).

### Fix — Strategy 1: Workspace Sync

```python
# Delete workspace → Frappe resyncs from JSON on same migrate run
try:
    frappe.delete_doc("Workspace", "TOC Buffer Management", force=True)
except Exception:
    # Fallback: direct SQL update on shortcuts
    frappe.db.sql("""
        UPDATE `tabWorkspace Shortcut`
        SET filters = REPLACE(filters, 'custom_toc_generated', 'custom_toc_recorded_by')
        WHERE parent = 'TOC Buffer Management'
    """)
```

### Fix — Strategy 2: Number Card Filters

```python
for nc in frappe.get_all("Number Card", filters={"module": "Chaizup Toc"}, fields=["name"]):
    filters = frappe.db.get_value("Number Card", nc.name, "filters_json") or ""
    if "custom_toc_generated" in filters:
        new_filters = (filters
            .replace('"custom_toc_generated"', '"custom_toc_recorded_by"')
            .replace('"=",1', '"=","By System"'))
        # Also apply date filter fix while we're here
        new_filters = new_filters.replace('"log_date","=","Today"', '"log_date","Timespan","today"')
        frappe.db.set_value("Number Card", nc.name, "filters_json", new_filters)
```

## fix_workspace_icon.py

### Problem
Workspace `TOC Buffer Management` icon was set to an SVG file path (`/assets/chaizup_toc/images/chaizup_toc_icon.svg`) during initial creation. Frappe's sidebar only renders **octicon names** (like `"graph"`, `"home"`, `"tools"`) — not file paths. The workspace appeared with a broken icon in the sidebar.

### Fix

```python
def execute():
    if frappe.db.exists("Workspace", "TOC Buffer Management"):
        icon = frappe.db.get_value("Workspace", "TOC Buffer Management", "icon")
        if icon and ("/" in icon or ".svg" in icon):
            frappe.db.set_value("Workspace", "TOC Buffer Management", "icon", "graph")
            frappe.db.commit()
```

Checks for the presence of `/` or `.svg` to identify file-path icons vs octicon names. Sets to `"graph"` — a valid Frappe octicon that represents charts/analytics, appropriate for a TOC dashboard.

## recompute_produced_qty_mirror.py (2026-05-25 — v0.0.10)

**Problem:** `Work Order.custom_produced_qty_in_uom` ("Manufactured Qty in UOM [TOC]") and `Production Plan Item.custom_produced_qty_in_uom` carried stale or zero values for many existing records, even though `produced_qty` and `custom_uom_conversion_factor` were both correct. Pre-patch sample: `MFG-WO-2026-00309` had produced_qty=5520, cf=60, but the mirror showed 78 (corresponding to a previous produced_qty=4680 snapshot, never re-synced). Several other WOs had mirror=0 despite produced_qty in the thousands.

**Two root causes:**

1. **Stock Entry hook missed entries.** `production_plan_engine.on_stock_entry_submit_refresh_produced` had a literal-name purpose check (`stock_entry_type in ("Manufacture", "Material Transfer for Manufacture")`). Sites with custom-named Stock Entry Types (e.g., "Manufacture - Plant A") failed the check, so the mirror refresh never fired. The nested `if purpose != "Material Transfer for Manufacture": return` made it even worse — early-returning for any purpose name that wasn't exactly one of the two literals. Same v0.0.10 commit rewrites the hook to refresh whenever `doc.work_order` is set (the only gate that actually matters).

2. **Install back-fill skipped populated rows.** `setup/install.py` updates WOs `WHERE custom_uom IS NULL OR custom_uom = ''`. Rows that already had a custom_uom but a stale mirror were never re-corrected on app update.

**Fix:** Raw SQL UPDATE on every Work Order and Production Plan Item with `custom_uom_conversion_factor > 0`:

```sql
UPDATE `tabWork Order`
   SET custom_produced_qty_in_uom = produced_qty / custom_uom_conversion_factor,
       custom_qty_in_uom          = qty          / custom_uom_conversion_factor
 WHERE custom_uom_conversion_factor > 0;
```

Scope notes (restricted areas):
- **No `docstatus` or `status` filter.** Stopped / Cancelled / Completed WOs also need correct mirrors (audit + ageing reports read them).
- **No `custom_uom IS NULL` skip.** This patch exists precisely to correct rows that already have a custom_uom.
- **Raw SQL, not `frappe.get_doc().save()`.** Calling save would fire validate() for thousands of rows and cascade-trigger TOC buffer recompute → minutes-long migrate. Raw UPDATE is fast and side-effect-free.
- **`update_modified = False` semantics implicit in raw SQL.** Preserves the audit trail of who last actually edited each WO.

**Verification:** Ran on chaizup-erp dev site, 2026-05-25. All 6 known-bad WOs corrected. Sweep query (`SELECT COUNT(*) WHERE ABS(expected - actual) > 0.0001`) returns 0 for both Work Order and Production Plan Item tables.

**Idempotent:** Re-running the patch on already-correct data is a no-op (same arithmetic on same data).

## sync_bom_list_view_settings.py (2026-05-27 — v0.0.11)

**Problem:** Operators viewing `/app/bom` saw the `name` (docname / "ID") column as the first visible column. Because BOM docnames look code-like (`BOM-FG-001-001`), the column was informally called "the workflow state column" — it visually resembled a status code without carrying product identity.

User request (2026-05-27): "in bom list fixtuare: Hide the workflowstate column, and make the status column in first column".

**Fix:** Reorder the BOM List View Settings fixture so the synthetic `{"type": "Status"}` column (which renders ERPNext's colored Active/Default/Template/Not-active pill from `get_indicator`) is at position 1. Drop `name` (ID) from the visible set — the Status pill + Item column together identify every row.

```python
# patches/v1_0/sync_bom_list_view_settings.py
import frappe

_BOM_LIST_VIEW_FIELDS = (
    '[{"type": "Status", "fieldname": "status_field", "label": "Status"}, '
    '{"fieldname": "item", "label": "Item To Manufacture"}, '
    # ... 4 more columns ...
)

def execute():
    if not frappe.db.exists("List View Settings", "BOM"):
        return  # fresh install — fixture importer will create it

                       _BOM_LIST_VIEW_FIELDS)
    frappe.db.commit()
```

**Why a patch was needed (not just the fixture change):** Frappe's `install_fixtures` is INSERT-only for existing rows. The BOM List View Settings row already existed on every site that had run earlier migrations — editing the fixture JSON alone was a silent no-op for those sites. The patch force-rewrites the `fields` JSON column.

**Restricted areas:**

- **Do not add a `workflow_state` Property Setter for BOM.** Verified 2026-05-27: BOM has no `workflow_state` DocField, Custom Field, Workflow attachment, or DB column. Adding a setter for a non-existent field would either no-op or raise on import.
- **Do not touch ERPNext's `get_indicator` callback in `bom_list.js`.** It controls the row-leading colored pill (Template / Default / Active / Not active) — that's the visual "Status" the user wants at the front. The synthetic `{"type": "Status"}` column in the fixture mirrors the same indicator into Report View; keep both in sync.
- **Do not remove `item` / `is_active` / `is_default` columns.** They're canonical BOM scanning columns; users filter on them daily.
- **Do not bypass this patch with a fixture-update fast path.** `install_fixtures` is INSERT-only by design.

**Verification (live, 2026-05-27):** Patch executed, `List View Settings.BOM.fields` rewritten. `bench migrate` registered the patch in `tabPatch Log` as `chaizup_toc.patches.v1_0.sync_bom_list_view_settings`. ## switch_wo_default_view_to_list.py (2026-05-27 — v0.0.13)

**Problem:** v0.0.12 shipped the 10-column List View Settings fixture + the combined "Work Order Actual Status" pill via `get_indicator`. **None of it rendered for end users** — they still saw the old 17-column Report view with duplicate "Created By", separate "Workflow State" + "Status" columns, and `Item Name`/`Stock UOM`/`Production Plan` extras.

**Root cause:** A Property Setter `Work Order-main-default_view = "Report"` (with `force_re_route_to_default_view = 1`) was force-routing every WO list URL to Report view. Report view:
- Ignores `List View Settings.fields` (the 10-column fixture)
- Ignores `frappe.listview_settings[dt].get_indicator` (the combined pill)
- Builds columns from `meta.fields where in_list_view=1` + `add_fields` + per-user `__UserSettings`

So all the v0.0.11/v0.0.12 list-view work was invisible.

**Fix:** Flip the Property Setter to `List` + clear per-user list-settings overrides (`tabDefaultValue` rows with `defkey LIKE "_list_settings:Work Order%"`, plus `tabUser Settings` rows on newer Frappe). Keeps `force_re_route_to_default_view = 1` so stale `?view=Report` URLs route back to List.

```python
# patches/v1_0/switch_wo_default_view_to_list.py
import frappe

def execute():
    frappe.db.set_value("Property Setter", "Work Order-main-default_view",
                       "value", "List")
    frappe.db.sql("""DELETE FROM `tabDefaultValue`
                      WHERE defkey LIKE '\\_list\\_settings:Work Order%'""")
    try:
        frappe.db.sql("DELETE FROM `tabUser Settings` WHERE doctype = 'Work Order'")
    except Exception:
        pass  # older Frappe — table doesn't exist
    frappe.clear_cache()
    frappe.db.commit()
```

**Restricted areas:**

- **Don't touch BOM's `default_view = Report`** — BOM intentionally uses Report view (sortable BOM navigator workflow).
- **Don't remove `force_re_route_to_default_view = 1`** — keeps stale `?view=Report` URLs from undoing the fix.
- **Don't widen the `__UserSettings`/`DefaultValue` clear to other doctypes** — per-user column picks elsewhere are legitimate user data.
- Re-running the patch is a no-op (`frappe.db.set_value` short-circuits identical writes; `frappe.db.delete` is filter-scoped).

**Verification (live, 2026-05-27):**
- Property Setter `Work Order-main-default_view = "List"` ✓
- `tabDefaultValue` rows with `_list_settings:Work Order%` defkey = 0 ✓
- `tabPatch Log` row for `chaizup_toc.patches.v1_0.switch_wo_default_view_to_list` ✓
- Bundle rebuilt via `bench build --app chaizup_toc`
- Users see 10-column List view with combined pill after browser hard-reload

**Lesson learned (added to fixtures.md INSERT-only gotcha section):** Changing list-view appearance requires verifying which VIEW is the default. List view honors fixtures + `get_indicator`; Report view ignores both. If the default is Report, any list-view-specific patch is invisible to users until you either flip the default OR target Report-view-specific config (Property Setter `in_list_view` flags + saved Report column orders).

NULL semantics are unavailable for Int — to distinguish "unknown" from "0 days" would require changing the field type to Data.

## wo_pp_filter_chips.py (2026-05-27 — v0.0.19)

**User spec:**
1. Add "Production Plan" filter chip to WO list
2. Add "Has Production Plan?" filter chip (with/without PP)

1. **Production Plan** — Property Setter `Work Order-production_plan-in_standard_filter = 1`. The existing standard `production_plan` Link field gets its own filter chip in the list header. Renders as an autocomplete search box.

```python
doc.custom_has_pp = "Yes" if (doc.get("production_plan") or "").strip() else "No"
```

## wo_drop_stale_item_group.py (2026-05-27 — v0.0.20)

**Root-cause correction of v0.0.15.** The original `custom_item_group` Custom Field used `fetch_from = "production_item.item_group"` — a SNAPSHOT at WO save time, not a live link. User correctly identified that reclassifying an Item's group later would silently leave old WOs filtering under the old group. **This is fundamentally inaccurate for a filter chip.**

**Filter accuracy principle (added to Claude memory as `feedback_filter_accuracy_principle.md`):**
> Filter chips, search fields, and any "filter by X" UX MUST resolve against the CURRENT state of the source-of-truth doctype at query time. Never back a filter with a fetch_from Custom Field, stored mirror, or cron-synced denormalization.

**This patch:**
1. Deletes the `Work Order-custom_item_group` Custom Field row.
2. Drops the SQL column `tabWork Order.custom_item_group` via `ALTER TABLE ... DROP COLUMN IF EXISTS` (with fallback for older MariaDB).
3. Clears the WO meta cache so the field disappears from the form/list immediately.

**Replacement (JS-side):** `public/js/work_order_list_extras.js` v0.0.20 adds a `setup_item_group_filter` handler in the listview's `onload`. Adds a toolbar button "Filter by Item Group" that opens a Frappe Dialog with a Link picker for Item Group. On selection, runs `frappe.db.get_list("Item", {filters: {item_group: X}})` LIVE (no cache) and applies the resulting item codes as `production_item in [...]` filter on the WO list. Zero drift possible — every click re-queries the live Item table.

**Restricted areas:**
- DO NOT re-add `custom_item_group` as any kind of stored field. The correctness requirement (filter = current state) is fundamentally incompatible with denormalized storage.
- DO NOT replace with an `is_virtual = 1` field. Virtual fields can't be filtered server-side via standard filter chips.
- DO NOT add a "sync on Item.validate" hook. Item.validate already does heavy work; fan-out updates to potentially thousands of WOs per Item save is dangerously slow at chaizup scale.
- DO NOT cache the item-code resolution in the JS. Each click must re-query so Item Group reclassifications propagate immediately.

**Verification (live, 2026-05-27):**
- `Custom Field.Work Order-custom_item_group` count = 0 ✓
- SQL column `tabWork Order.custom_item_group` count = 0 ✓
- Patch registered in `tabPatch Log` ✓
