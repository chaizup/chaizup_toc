# `tasks/` — Scheduled Background Tasks

## Purpose
All scheduled job functions. Called by the Frappe scheduler via `hooks.py → scheduler_events`. Each function is a top-level module function (not a class method).

---

## `daily_tasks.py`

### Daily Rhythm (all times IST)

| Time | Function | What It Does |
|------|----------|--------------|
| 06:30 AM | `daily_adu_update()` | Auto-calculate ADU for all TOC items |
| 07:00 AM | `daily_production_run()` | Calculate buffers + generate MRs for all types |
| 07:30 AM | `daily_procurement_run()` | Log warning for RM/PM Red zone items only |
| 08:00 AM | `daily_buffer_snapshot()` | Archive all buffer states to TOC Buffer Log |
| 08:00 AM Sunday | `weekly_dbm_check()` | DBM auto-adjust (TMR/TMG) |

---

### `daily_adu_update()` — 06:30 AM

Calculates Average Daily Usage (ADU) for all enabled TOC items.

**R1**: If `custom_toc_custom_adu = 1` → **skips that item entirely**. User-entered manual ADU is preserved.

**Calculation by buffer type:**

| Type | SQL Source | Logic |
|------|-----------|-------|
| FG | `Delivery Note Item` JOIN `Delivery Note` | `SUM(qty)` WHERE `dn.docstatus=1` in period |
| RM/PM | `Stock Entry Detail` JOIN `Stock Entry` | `SUM(qty)` WHERE `se.stock_entry_type IN ('Material Issue', 'Manufacture', 'Material Transfer for Manufacture')` AND `sed.s_warehouse IS NOT NULL` |
| SFG | `Stock Entry Detail` JOIN `Stock Entry` | `SUM(qty)` WHERE `sed.s_warehouse IS NOT NULL` (all Stock Entry types) |

**Period options** (from `custom_toc_adu_period`):
- "Last 30 Days" → 30 days
- "Last 90 Days" → 90 days (default)
- "Last 180 Days" → 180 days
- "Last 365 Days" → 365 days

**ADU formula**: `total_qty / number_of_days_in_period`

**Writes to**: `Item.custom_toc_adu_value` and `Item.custom_toc_adu_last_updated` (with `update_modified=False`).

**Error handling**: Per-item exceptions are logged to Error Log but the batch continues. A single outer try/except logs catastrophic failures.

---

### `daily_production_run()` — 07:00 AM

Calls `generate_material_requests()` with no filters (all buffer types, all zones per settings).

This generates MRs for **all types** (FG, SFG, RM, PM) based on `TOC Settings.mr_zones` and `auto_generate_mr` flag.

---

### `daily_procurement_run()` — 07:30 AM

Calculates RM + PM buffers and logs a WARNING to the scheduler log if any items are in Red/Black.

> **Important**: This does NOT generate Material Requests. MRs for RM/PM are generated by `daily_production_run()` at 07:00 AM (no buffer_type filter). The procurement run is a monitoring/alerting step only.

---

### `daily_buffer_snapshot()` — 08:00 AM

Archives all buffer states to `TOC Buffer Log`:
1. Calls `calculate_all_buffers()` (all types).
2. For each buffer result, creates a `TOC Buffer Log` document with all F2/F3 values.
3. Inserts with `ignore_permissions=True`.
4. Per-log failures are silently swallowed (continues to next).
5. Commits once at end.

This data feeds the DBM engine's TMR/TMG logic.

---

### `weekly_dbm_check()` — 08:00 AM Sunday

Delegates to `dbm_engine.evaluate_all_dbm()`. See `toc_engine/toc_engine.md` for DBM logic.

## Bug History

### ~~BUG (FIXED): `daily_procurement_run()` misleadingly documented~~

The `hooks.py` scheduler comment previously said "generates Purchase MRs" for the 07:30 AM task, which was wrong. `daily_procurement_run()` only calls `frappe.logger().warning()` for Red/Black RM/PM items — no MRs are created.

**Fixed**: Both `hooks.py` scheduler comment and `daily_procurement_run()` docstring updated to accurately say "monitoring-only — no MRs created here". MRs for RM/PM are generated by `daily_production_run()` at 07:00 AM (which has no buffer_type filter).
